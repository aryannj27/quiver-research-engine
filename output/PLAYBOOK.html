Prerequisites

One-time setup. Ten minutes.

  1. Clone the repo. pip install -r requirements.txt.
  2. Create a .env file from .env.example. Populate:
    • QUIVER_API_KEY — from the Quiver dashboard.
    • ANTHROPIC_API_KEY — for the narration step. Sonnet 4.6 is the model.
  3. Source the env: export $(cat .env | xargs).
  4. Dry run: python run.py. Expected runtime ~90 seconds.
Checking you're set up right
On a successful dry run, you should see lines ending in ✓ Open: output/sunday_brief_YYYY-MM-DD.html. Open that file in any browser. If you see at least one ranked lead card, you are done.

Sunday Evening — The 60-Minute Sprint

Every Sunday, 6:00 PM — 7:00 PM ET. One hour, four commands, one brief.

1
Pull the data
6:00 PM · ~40s
# From the repo root:
python pull.py

This hits five endpoint families:

EndpointWhat you getFlakiness
/beta/live/congresstrading~200 most recent congressional trades with ExcessReturn baked inReliable
/beta/live/offexchange~500 tickers with daily OTC volume and DPI (dark pool index)Reliable
/beta/live/insiders?ticker=XExecutive / director Form 4 transactions for a single tickerFlaky. Bulk endpoint has DB timeouts; per-ticker works.
/beta/historical/lobbying/XQuarterly lobbying filings with specific issues lobbiedReliable per-ticker
/beta/historical/govcontracts/XQuarterly federal contract rollups for a single tickerReliable per-ticker

The pull script auto-picks the top 15 tickers by congressional activity and enriches only those — this keeps runtime under 2 minutes and well under rate limits.

If the pull fails
Empty congress data: Check curl -H "Authorization: Token $QUIVER_API_KEY" https://api.quiverquant.com/beta/live/congresstrading | head. If it returns HTML, your key is likely invalid or rate-limited. Wait 5 min and retry.
QueuePool / SQLAlchemy errors: That's a Quiver backend hiccup. The script logs and skips. You will see [quiver-db-timeout] in stderr. Proceed anyway — the pipeline degrades gracefully.
2
Run the scorer
6:15 PM · ~5s
python score.py

This runs seven deterministic rules over the pulled data. Output is a ranked JSON list of leads in data/leads_latest.json plus a terminal preview of the top 10. A good Sunday produces 5–15 leads total; 2–5 make it to the final narrated brief.

What counts as a "good" Sunday
At least one lead scored 80+ = there's a real story for Monday. No leads above 70 = it's a quiet week (happens). See the empty-brief protocol.
3
Claude narration
6:30 PM · ~20s · ~$0.05
python narrate.py

This sends the top 10 ranked leads to Claude Sonnet 4.6 with the Quiver editorial voice prompt (see prompts section). Claude returns, for each lead:

  • pitch — one sentence, Pelosi Principle compressed
  • angle — one paragraph, why this matters
  • verify — 2–3 concrete verification steps
  • headline — 3 draft headlines, punchy, <10 words each
  • confidence — high / medium / low

Important: Claude never sees raw anomaly-detection data. It only sees pre-scored, pre-verified signals. Claude writes. Python thinks. This split is the whole point.

4
Verify each lead (the human-in-the-loop step)
6:40 PM · ~15 min

The system can surface a lead but cannot publish one. For each lead in the top 5, spend 2–3 minutes doing the following manual checks:

  1. Confirm the disclosure. Search the member's name on the House clerk site (disclosures-clerk.house.gov) or the Senate eFD portal. Quiver can occasionally be stale by a day or two.
  2. Check committee assignments. Does the member actually sit on a committee with jurisdiction over the traded ticker's sector? clerk.house.gov/Members or senate.gov/general/committee_assignments.
  3. Look for the catalyst. Is there a pending vote, markup, or hearing related to the ticker's industry? Google: "[ticker industry] committee markup [current month]".
  4. Check for dupe coverage. Has Bloomberg, Reuters, or another newsletter already published this? Search "[member name] [ticker]" in Google News, past week.
Pro tip
A lead that passes all four checks is publishable. A lead that fails #4 (already covered elsewhere) drops to the "mention" pile — reference it in the Tuesday newsletter with an original angle, don't lead with it.
5
Package for Monday
7:00 PM · ~2 min
python brief.py

Renders the final self-contained HTML brief at output/sunday_brief_YYYY-MM-DD.html. Forward this file (or paste-to-Notion) to David. Done.

Or just run the whole pipeline end-to-end with python run.py. Single command. Same result.

The Scoring Decision Tree

How the scorer decides what to flag. All thresholds tunable in score.py.

Is it a congressional options trade ≥ $15K? (R1)
Yes → score 55 + dollar-magnitude boost + profile boost. Auto-flag.
Is it a congressional stock trade ≥ $100K? (R2)
Yes → score 40 + dollar boost + excess-return boost.
Did the same member trade ticker X within 90 days of lobbying activity on X? (R3)
Yes → score 70 + profile boost. This is the highest-signal single rule.
Did the member trade ticker X in same/adjacent quarter to a 1.5×+ spike in X's federal contracts? (R4)
Yes → score 75 + profile boost + large-contract boost.
Did ≥ 3 members trade the same ticker in the same direction within 14 days? (R5)
Yes → score 50 + member-count boost + bipartisan boost. Bipartisan clusters are highest priority.
Does a ticker have elevated dark-pool short rate (DPI > 0.55) AND congressional activity? (R6)
Yes → score 45 + DPI-magnitude boost.
Are insiders net-selling $10M+ in 30 days on a ticker Congress is also trading? (R7)
Yes → score 60 + dollar-magnitude boost. Strongest when congress is also selling.

The "name recognizability" boost

Any lead featuring one of ~20 high-profile names gets a +15 bump. The current list is in score.py → HIGH_PROFILE and includes Pelosi, Gottheimer, Crenshaw, Tuberville, Warren, Cruz, Schumer, McConnell, and others. This is editable. The list reflects who David's readers recognize, not who is most active.

The Actual Prompts

Copy-pasteable. Already in narrate.py. Included here so David can tune the voice without digging into code.

System PromptYou write for Quiver Quantitative, a financial newsletter read by 317,000+ retail investors. The voice: data-first, punchy, conversational. Think Morning Brew meets a Bloomberg terminal. Every story follows the "Pelosi Principle": a clear NAME + CONFLICT + DOLLAR AMOUNT. You are the analyst behind the analyst — you do NOT write final stories. You produce short research briefs for the Head of Sales, David Love, so he can decide which leads to turn into full stories on Monday morning. For each lead you receive, produce a JSON object with these fields: - "pitch": one sentence (≤25 words). The hook. Pelosi Principle compressed. - "angle": one paragraph (3-4 sentences). Why this matters. What to investigate. - "verify": 2-3 concrete verification steps (committee lookup, SEC filing check, etc). - "headline": 3 draft headline options, each ≤10 words, punchy. - "confidence": "high" | "medium" | "low" — your editorial confidence this is a real story. Output ONLY a JSON array of these objects, in the same order as input. No preamble, no markdown fences.

Why few-shot. The prompt includes one example lead (a hypothetical Pelosi NVDA options trade) and its ideal narration. This anchors Claude to the Quiver voice within the first response — no warm-up turns needed. The example is fictional (clearly marked) so Claude cannot hallucinate it as fact in the output.

Why JSON output. Forces structured output so brief.py can render without parsing natural-language ambiguity. Saves David from having to copy-edit Claude's prose — each field lands in a specific slot in the HTML brief.

Why Sonnet 4.6, not Opus. For narrating 10 already-scored leads, Sonnet is fast enough (sub-20s) and accurate enough. Opus is overkill and 5× the cost. Haiku is too loose on voice. Sonnet is the right tier.

Tunable knob. Change TOP_N in narrate.py to narrate more or fewer leads. More leads = more Claude tokens = slightly higher cost. Recommend keeping at 10; below that you lose range, above you lose signal-to-noise.

What Didn't Work (And Why)

Rules I prototyped and dropped. Included for epistemic transparency — if David or a future intern is tempted to re-add these, here is why I didn't.

Dropped: "Trade timing vs earnings date" rule

I tried flagging congressional trades within ±7 days of earnings. The intuition was that trading immediately pre-earnings signals inside information. In practice, this rule surfaced mostly noise: ~60% of S&P 500 tickers are within ±7 days of earnings at any given moment, so essentially every congressional trade in a major ticker triggered. Signal died. Takeaway: temporal rules need tighter joins (earnings + committee assignment + same direction as analyst revisions), not just proximity.

Dropped: "Patent grant delta" rule

Quiver has a patents endpoint with momentum metrics. I considered flagging companies with sudden patent-grant spikes as story fodder. Dropped because patent activity is interesting but not structured around the "name + conflict + dollar" triple. It's a story type for a different newsletter segment (emerging tech, not political-finance overlap). Noted in Extend.

Dropped: "LLM-as-anomaly-detector" experiment

Early design: feed raw congressional trade JSON to Claude and ask "find the anomalies." This is the approach David explicitly warned against in the brief, and rightly so. Claude is fantastic at narrating a pre-identified anomaly and terrible at identifying one. The LLM surfaced plausible-sounding but unverifiable patterns ("this pattern suggests coordinated activity...") — confabulation, not analysis. The hard split between deterministic scoring and LLM narration is the single most important design decision in this system.

What to Extend Next

Features I deliberately deferred. Roughly ordered by ROI.

  1. 13F position-change signals. Cross-reference congressional buys with institutional accumulation. High signal, 1 day of work.
  2. Wikipedia traffic spike detector. Wikipedia view counts for ticker pages sometimes precede earnings surprises. Meta-interesting, 2 days.
  3. Historical lead backtest. Run the scorer over the past 12 months; check whether high-scored leads actually became Quiver newsletter stories. Calibrates the thresholds. 3 days.
  4. Conflict Web as per-lead image embed. Auto-generate a PNG of the Conflict Web for every lead scored 80+, embed directly in the HTML brief. Makes the brief self-illustrating. 1 day.
  5. Slack bot delivery. Instead of emailing an HTML file, post top 3 leads to a #sunday-brief Slack channel every Sunday at 7 PM. 0.5 days.
  6. Patent & consumer app data. Pull in Quiver's patent and app-rating endpoints for tech/consumer story angles. Supplementary, not core.

When the System Finds Nothing

Some Sundays the scorer returns no lead above score 70. This is a feature, not a bug.

Quiet weeks exist. Congress recesses, filings lag, insiders go silent. A research engine that manufactures excitement on quiet weeks is worse than one that honestly reports the absence of signal.

Protocol:

  1. Re-run the pull — sometimes Quiver backfills disclosures by Monday morning.
  2. Lower the scoring threshold temporarily: change TOP_N = 10 to TOP_N = 15 in narrate.py, re-render brief.
  3. If still nothing above 60 — send David the brief with a note: "Low-signal week. Top-3 items attached. Consider running a non-anomaly story (retrospective, educational, or industry deep dive)."
  4. Log the quiet week in data/quiet_log.md. If 3+ quiet weeks in a row, the rules likely need retuning — too restrictive.
What not to do
Do not lower the score threshold and pretend a mediocre lead is a strong one. David's readers can tell. The research engine's credibility comes from the weeks it flags real stories — which requires the discipline to flag nothing on the weeks there is nothing.
